1. Executive Summary

PeerLink Mobile is a React Native application that enables two mobile devices to establish a direct, encrypted, peer-to-peer communication channel over the internet Ś without routing user messages or files through any central server. Leveraging WebRTC DataChannels, PeerLink delivers near-native network speeds for both messaging and large file transfers across any network topology, including cellular, Wi-Fi, and mixed-carrier environments.

The core value proposition is simplicity, speed, and privacy: once two devices connect, every byte transferred between them is encrypted end-to-end by the DTLS specification and never touches a relay server. The minimal signaling backend is used only for connection bootstrapping and carries zero user data.

1.1 Problem Statement
Existing messaging apps route all data through proprietary central servers Ś creating latency, limiting transfer sizes, and introducing a third-party data custodian. When two users want to transfer a 2 GB video file, current solutions either reject it, compress it, or take minutes. PeerLink removes the server from the data path entirely.
1.2 Solution
Ģ Direct P2P connection between two devices using WebRTC DataChannel
Ģ Signaling-only backend (Node.js + Socket.IO) Ś handles handshake only, never user data
Ģ STUN for NAT traversal; TURN relay fallback only when P2P path is impossible
Ģ Binary file chunking with progress tracking, resume support, and integrity checking
Ģ Text messages delivered at sub-100ms latency on good connections

5.2 Connection Establishment Flow
The following sequence describes the complete connection establishment between Device A (initiator) and Device B (joiner):

Ģ Step 1 Ś Room creation: Device A requests a room from signaling server, receives a 6-char alphanumeric code and joins the Socket.IO room.
Ģ Step 2 Ś Room join: Device B enters the code; signaling server emits a 'peer-joined' event to both parties.
Ģ Step 3 Ś SDP Offer: Device A creates an RTCPeerConnection, creates a DataChannel, calls createOffer(), sets local description, sends SDP offer via signaling server.
Ģ Step 4 Ś SDP Answer: Device B receives offer, sets remote description, calls createAnswer(), sets local description, sends SDP answer via signaling server.
Ģ Step 5 Ś ICE Exchange: Both devices emit ICE candidates as they are gathered; each sets received candidates on the peer connection via addIceCandidate().
Ģ Step 6 Ś P2P Attempt: ICE agent attempts direct connection (host candidates), then reflexive (STUN), then relay (TURN) in order of priority.
Ģ Step 7 Ś DataChannel Open: Once ICE completes, the DataChannel fires 'open' event. Both devices update UI to 'Connected'. Signaling server connection is no longer needed and can be dropped.

5.3 File Transfer Protocol
Files are transferred as a sequence of binary chunks over the DataChannel with a control protocol layered in JSON messages:

Ģ TRANSFER_INIT: sender emits { type, fileId, fileName, fileSize, mimeType, chunkSize, totalChunks, sha256 }
Ģ TRANSFER_ACCEPT / TRANSFER_REJECT: receiver responds with acceptance or rejection (e.g. insufficient storage)
Ģ CHUNK: sender emits ArrayBuffer prefixed with 8-byte header { fileId (4B), chunkIndex (4B) }
Ģ CHUNK_ACK: receiver acknowledges every N chunks (configurable, default: every 16 chunks) for flow control
Ģ TRANSFER_COMPLETE: sender emits after final chunk; receiver computes SHA-256 and responds with VERIFY_OK or VERIFY_FAIL
Ģ TRANSFER_PAUSE / TRANSFER_RESUME: either party can pause; resume includes last ACKed chunk index
Ģ TRANSFER_ABORT: either party cancels; receiver deletes partial file

Chunk size is dynamically tuned: starts at 16 KB, increases to 64 KB if throughput is stable, reduces on buffer pressure. The DataChannel bufferedAmountLowThreshold is used to gate chunk emission and prevent memory overflow on slow links.

6.2 Connection States
Ģ IDLE: No active session, home screen
Ģ WAITING: Room created, awaiting peer to join (showing room code)
Ģ NEGOTIATING: SDP offer/answer and ICE exchange in progress
Ģ CONNECTED_P2P: Direct DataChannel open, P2P path confirmed
Ģ CONNECTED_RELAY: DataChannel open via TURN relay
Ģ RECONNECTING: Connection dropped, ICE restart in progress
Ģ FAILED: All ICE candidates exhausted, manual retry required
Ģ CLOSED: User-initiated disconnect

7.2 Security & Privacy
Ģ All DataChannel traffic is encrypted using DTLS 1.2 or 1.3 as mandated by the WebRTC specification Ś no exceptions, no opt-out.
Ģ The signaling server must not log, store, or inspect SDP payloads beyond routing them to the correct room. Rooms are automatically deleted from memory 60 seconds after both peers disconnect.
Ģ TURN server credentials must be time-limited (TTL 24h) and generated per-session using HMAC-SHA1 as per RFC 5766.
Ģ Room codes must have sufficient entropy to prevent brute-force joining: 6 alphanumeric characters (case-sensitive) yields 56 billion combinations; rooms expire after 10 minutes of inactivity.
Ģ File integrity is verified with SHA-256 after every transfer. The hash is computed client-side and transmitted in the TRANSFER_INIT control message.
Ģ No analytics, telemetry, or crash reporting that includes message content or file metadata is permitted.
Ģ The app must not request location, contacts, or camera permissions. Only storage and microphone (future) are permissible.

7.3 Reliability
Ģ ICE restart must be attempted automatically on connection drop before surfacing an error to the user.
Ģ File transfers interrupted by a connection drop must be resumable from the last ACKed chunk Ś no full restart.
Ģ The app must handle background / foreground transitions without losing connection state (iOS background modes: voip socket keepalive).
Ģ The signaling server must be horizontally scalable with Redis pub/sub for room state (Socket.IO Redis adapter).

7.4 Accessibility
Ģ All interactive elements must meet WCAG 2.1 AA touch target size (44x44pt minimum).
Ģ VoiceOver (iOS) and TalkBack (Android) must announce message receipt, connection status changes, and transfer progress.
Ģ Color is never the sole indicator of status Ś icons and labels supplement all color-coded states.
Ģ System font size preferences must be respected (Dynamic Type on iOS, font scale on Android).

8.2 Deployment Requirements
Ģ Runtime: Node.js 20 LTS
Ģ Framework: Express 4 + Socket.IO 4 with Redis adapter for multi-instance scaling
Ģ Hosting: any platform supporting persistent WebSocket connections (Railway, Fly.io, Render, or self-hosted)
Ģ SSL: required Ś WebRTC will not negotiate over non-secure WebSocket (ws://) in production
Ģ Room cleanup: in-memory rooms purged after 10 minutes of inactivity or when both peers disconnect
Ģ Rate limiting: max 10 room creations per IP per minute; max 30 ICE candidate relays per second per room
Ģ Health endpoint: GET /health returns { status: 'ok', rooms: N, connections: N }

12. Glossary

TermDefinitionWebRTCWeb Real-Time Communication Ś W3C/IETF standard for browser and native P2P audio, video, and dataDataChannelThe WebRTC API for arbitrary binary or text data transfer between peers, independent of media streamsSDPSession Description Protocol Ś text format describing media/data capabilities used in WebRTC offer/answer negotiationICEInteractive Connectivity Establishment Ś RFC 8445 process for finding the best network path between two peersSTUNSession Traversal Utilities for NAT Ś server that helps a device discover its public IP and portTURNTraversal Using Relays around NAT Ś relay server used when direct P2P is impossible due to strict NAT/firewallDTLSDatagram Transport Layer Security Ś the encryption protocol used to secure WebRTC DataChannel connectionsNATNetwork Address Translation Ś router mechanism that maps private IPs to a shared public IP; the primary P2P obstacleSignaling ServerLightweight server used only to exchange WebRTC handshake messages; never sees user dataChunkA fixed-size binary fragment of a file used in chunked file transfer to avoid memory overflowSHA-256Secure Hash Algorithm 256-bit Ś cryptographic hash used to verify file integrity after transferP2PPeer-to-peer Ś direct device-to-device connection with no server in the data pathICE RestartRe-initiating the ICE negotiation on an existing connection to recover from a network change or drop
